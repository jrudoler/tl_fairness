# Experiment Reproduction

The paper is reproduced by a Snakemake workflow. Raw inputs land in `data/raw/`,
simulation/analysis intermediates in `data/generated/`, and final figures and
tables in `results/`. See the `Snakefile` for the full dependency graph and the
`science-repo.md` skill for the layout rationale.

## Setup

```bash
uv sync
```

## Run the workflow

```bash
uv run snakemake --cores 1            # build all figures and tables
uv run snakemake --cores 1 -n         # dry run: show the plan without executing
uv run snakemake --cores 1 --forcerun sim_robust   # rebuild one (protected) rule
uv run snakemake --cores 1 validate   # validate generated artifacts
```

During development you can run a single step directly (Snakemake sets these for
you, but this is handy for debugging):

```bash
PYTHONPATH=. uv run python analysis/sim_robust/run.py --output data/generated/robust_res.csv
```

Expensive outputs (`sim_robust`, `sim_cmi`, `analyze_adult`, `analyze_law`) are
marked `protected()` so they are not regenerated by accident; use `--forcerun`
to rebuild them deliberately.

## Parallel execution (expensive rules)

`analyze_adult`, `analyze_law`, and `sim_cmi` are embarrassingly parallel and
run serially by default. On a multi-core machine, request cores and matching
workers with `--config njobs=<N>`:

```bash
# one fat node, e.g. 64 cores: each expensive rule uses all of them, one rule
# at a time (threads: njobs keeps Snakemake from co-scheduling them)
uv run snakemake --cores 64 --config njobs=64 analyze_adult
uv run snakemake --cores 64 --config njobs=64 analyze_law
uv run snakemake --cores 64 --config njobs=64 sim_cmi
```

What parallelizes, and the guarantees:

- **Permutation importance** (`analyze_*`): each distinct feature-subset fit is
  independent and depends only on the subset + the estimator's fixed
  `random_state`, so the parallel result is **bit-identical** to the serial run.
  Near-linear speedup (~11 h to ~20 min for Adult on 32-64 cores).
- **CMI simulations** (`sim_cmi`): each simulation gets an independent RNG via
  `Generator.spawn()`, so results are **reproducible from `--seed` and invariant
  to the worker count** (`njobs=2` and `njobs=64` give identical numbers). The
  serial and parallel paths draw different streams, but each is reproducible.
- `sim_cmi` also caps the *comparison* step's workers so concurrent `knncmi`
  distance arrays (~4 GB each at n=10000) fit in `--config compare_mem_gb=<GB>`
  (default 16). The coverage step has no such footprint and uses all workers.

GPU does not help here: the estimators are CPU-only sklearn tree ensembles
(`HistGradientBoostingClassifier`, `GradientBoostingClassifier`) and a numpy
`knncmi`; Apple-Silicon/MPS is irrelevant to numpy/sklearn. The win is CPU
parallelism. For a cluster, each `(metric)` / `(c, size)` unit is independent
and could be split into per-unit Snakemake jobs behind a SLURM executor.

Reproducibility note: seeding the CMI estimator (`GradientBoostingClassifier`)
also fixed a latent bug where it drew from the unseeded global numpy singleton,
making the original CMI sims vary run-to-run even serially.

Note on Figure 2: the row labels read well-specified -> fully misspecified, with
labels derived from the case names so they stay aligned with the data. (The
published figure had the top/bottom rows swapped; see `analysis/fig2_robust/run.py`.)

Note on Table 2: `sim_cmi` defaults to the paper-compatible Monte Carlo estimand
that matches Appendix A. Pass `--conditional-truth` to compute the literal
conditional CMI target instead.

The reusable simulation/estimator code lives in `tlfair/` (`metrics.py`,
`superlearner.py`, `knncmi.py`, `simulations.py`, `cmi_sim.py`, `plotting.py`).
The `analysis/<rule>/run.py` scripts are thin entrypoints that parse arguments,
read declared inputs, call `tlfair`, and write declared outputs. The notebooks
under `notebooks/` are exploratory and are no longer the source of truth for any
figure.

## Runtime Notes

Observed on this machine:

```text
Law permutation importance, 1 permutation / 10 variables: ~21.5s
Adult permutation importance, 1 permutation / 11 variables: ~42.0s
Robust probe, n=250 and 2 reps: ~0.4s for the simulation loop
CMI probe, n=500 and 2 coverage sims: ~1.0s per c value
```

Measured paper-size runtime (`--cache-importance`, default `n_samples=1000`):

```text
Law data analysis,  8 cores (--config njobs=8): ~4.9 hours  (cmi metric ~3.7h of it)
Adult data analysis, 8 cores: ~12-20 hours (extrapolated; ~2x Law's subsets on larger data)
Adult/Law, ~64-core node: roughly 2-3 hours
sim_cmi, 8 cores: ~45-60 minutes (coverage ~30 min; compare adds the knncmi pass)
Robust simulation: ~5-10 minutes (single-threaded)
```

The data-analysis cost is dominated by the **cmi importance metric**: it wraps
the booster in a 3-fold `CalibratedClassifierCV`, so each feature-subset is
~3 fits. With caching, Adult explores up to 2,047 subsets per metric and Law up
to 1,023, single-threaded per worker (~100s/subset for cmi on Law). Serial, the
cmi metric alone is ~28h on Law -- the `--config njobs` parallelism is what makes
these tractable, and scales near-linearly with cores. For Adult, prefer a
many-core node; `--importance-samples` trades importance precision for speed.
